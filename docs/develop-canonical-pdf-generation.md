# Develop-Canonical PDF Generation Pattern

A practical guide based on solving recurring binary merge conflicts when promoting documentation PDFs across branches (develop→staging→main). This pattern ensures PDFs are generated once on develop and flow forward cleanly through promotions.

## The Problem

Projects that auto-generate PDFs from markdown documentation hit the same issue: binary PDFs cause unresolvable merge conflicts on every branch promotion. Git cannot merge binary files, so if a PDF is regenerated independently on multiple branches, every promotion PR shows a conflict.

### What We Tried (and Why It Failed)

1. **PDFs tracked on all branches, generated everywhere** — Merge conflicts on every promotion because each branch had a slightly different binary
2. **PDFs removed from git entirely** — No conflicts, but PDFs weren't available on staging/main for users who needed them
3. **`.gitignore` block with exceptions** — Contradictory rules (duplicate `*.pdf` lines, re-allow exceptions) created confusion about what was actually tracked

### Gotcha #1: Duplicate `.gitignore` Rules

Having `*.pdf` appear twice in `.gitignore` with `!` exceptions scattered between them leads to unpredictable behavior. Gitignore processes rules top-to-bottom — later rules override earlier ones, but it's easy to lose track when rules are duplicated across sections.

**Fix**: One `*.pdf` line, followed immediately by all `!` exceptions.

## The Solution: Develop-Canonical PDFs

PDFs are generated **only on feature branches** (via pre-commit hook) and flow forward through promotions. Staging and main never independently generate PDFs.

```
feature/* → develop → staging → main
  ↑ generated here    (flow forward only, never regenerated)
```

This eliminates conflicts because merge conflicts only occur when **both sides** modify the same file. If only the develop side ever modifies PDFs, promotions are always one-sided — clean merges.

## Implementation

### Step 1: PDF Generator Script

Create a generator that converts markdown sources to PDFs. Support a `--only` flag for selective generation (so the pre-commit hook only regenerates PDFs for changed docs, not all of them).

```typescript
// src/services/generate-docs-pdf.ts
const DOC_MAPPINGS = [
  { source: 'README.md', pdf: 'README.pdf', title: 'Project Documentation' },
  { source: 'RELEASE_NOTES.md', pdf: 'RELEASE_NOTES.pdf', title: 'Release Notes' },
  { source: 'docs/DESIGN.md', pdf: 'docs/DESIGN.pdf', title: 'Architecture' },
  { source: 'docs/SECURITY.md', pdf: 'docs/SECURITY.pdf', title: 'Security' },
  // Add all markdown→PDF mappings
];

// Parse --only flag: npm run generate-docs-pdf -- --only README.md docs/DESIGN.md
function parseOnlyFilter(): string[] | null {
  const onlyIdx = process.argv.indexOf('--only');
  if (onlyIdx === -1) return null;
  return process.argv.slice(onlyIdx + 1).filter(a => !a.startsWith('-'));
}
```

Add an npm script:
```json
{
  "scripts": {
    "generate-docs-pdf": "npm run build:js && node dist/services/generate-docs-pdf.js"
  }
}
```

### Gotcha #2: Missing PDFs in the Mapping

When adding new markdown documentation (e.g., `docs/SECURITY.md`), remember to add the mapping in **both** the generator script and the pre-commit hook's `git add` list. We missed `docs/SECURITY.pdf` in the pre-commit hook for months — it was generated by the script but never staged.

### Step 2: `.gitignore` Configuration

Block all PDFs globally, then whitelist only the doc PDFs that should be tracked:

```gitignore
# PDFs — ignore all except doc PDFs auto-generated by pre-commit hook on develop
# Doc PDFs are canonical on develop and flow forward via promotions to staging/main
*.pdf
!README.pdf
!RELEASE_NOTES.pdf
!docs/*.pdf
```

**Rules:**
- `*.pdf` must come **before** the `!` exceptions (gitignore is order-dependent)
- Only **one** `*.pdf` line — duplicates cause confusion
- Only whitelist PDFs generated from tracked markdown sources
- Application-generated PDFs (reports, exports) should remain ignored

### Gotcha #3: Force-Add After Changing `.gitignore`

If PDFs were previously removed from git (via `git rm`), changing `.gitignore` to allow them isn't enough — you must `git add -f` to re-track them:

```bash
npm run generate-docs-pdf
git add -f README.pdf RELEASE_NOTES.pdf docs/*.pdf
```

The `-f` flag overrides `.gitignore` for the initial add. Once tracked, subsequent changes are picked up normally.

### Step 3: Pre-Commit Hook

Add PDF generation to your pre-commit hook (e.g., `.husky/pre-commit`), guarded to run **only on feature branches**:

```bash
# Auto-generate doc PDFs when source markdown files are staged
# Only on feature branches — PDFs flow forward via promotions to staging/main
CURRENT_BRANCH=$(git rev-parse --abbrev-ref HEAD)
DOC_MD_STAGED=$(git diff --cached --name-only | \
  grep -E '^(README\.md|RELEASE_NOTES\.md|docs/(DESIGN|INFRASTRUCTURE|DEVELOPMENT|SECURITY)\.md)$' || true)

if [ -n "$DOC_MD_STAGED" ] && \
   [ "$CURRENT_BRANCH" != "develop" ] && \
   [ "$CURRENT_BRANCH" != "staging" ] && \
   [ "$CURRENT_BRANCH" != "main" ]; then
  echo "📄 Markdown docs staged — regenerating PDFs..."
  npm run build:js --silent 2>/dev/null
  if npm run generate-docs-pdf --silent -- --only $DOC_MD_STAGED 2>/dev/null; then
    git add README.pdf RELEASE_NOTES.pdf \
          docs/DESIGN.pdf docs/INFRASTRUCTURE.pdf \
          docs/DEVELOPMENT.pdf docs/SECURITY.pdf 2>/dev/null || true
    echo "✅ Doc PDFs regenerated and staged"
  else
    echo "⚠️  PDF generation failed — skipping (PDFs may be stale)"
  fi
fi
```

### Gotcha #4: Why Feature Branches, Not Develop?

The hook guards against `develop`, `staging`, and `main` — but you might wonder why not generate on develop too. The answer: **you don't commit directly on develop**. Commits arrive on develop via PR merges from feature branches. The PR already contains the regenerated PDFs from the feature branch commit.

If you allowed generation on develop, it would only fire during merge commits (which you don't control) and could cause issues with automated processes.

### Gotcha #5: Non-Blocking Failure

PDF generation depends on Puppeteer/Chrome being available. In CI environments or fresh clones without Chrome, generation will fail. The hook wraps the generation in an `if` block so failure produces a warning but **does not abort the commit**. Stale PDFs are better than blocked commits.

### Step 4: Initial Setup for Existing Repos

If your repo already has PDFs on some branches but not others:

**Scenario A: PDFs exist on staging/main but were removed from develop**
- The promotion merge will show `modify/delete` conflicts
- Resolve by re-adding PDFs to develop first (this guide), then promoting

**Scenario B: PDFs never existed on any branch**
```bash
# On a feature branch:
npm run generate-docs-pdf
git add -f README.pdf RELEASE_NOTES.pdf docs/*.pdf
git commit -m "feat: Add doc PDFs with develop-canonical pattern"
# Merge to develop, then promote normally
```

### Gotcha #6: Promotion After Initial Setup

The first promotion after setting up this pattern may still conflict if staging/main have old PDFs that differ from develop's newly generated ones. This is a one-time issue — resolve by taking develop's version:

```bash
# During the conflicted merge:
git checkout --theirs README.pdf RELEASE_NOTES.pdf docs/*.pdf
git add README.pdf RELEASE_NOTES.pdf docs/*.pdf
git commit
```

After this single resolution, all subsequent promotions will be clean.

## Verification

Write tests to ensure the pattern stays intact:

```typescript
describe('PDF generation pattern', () => {
  test('pre-commit hook generates PDFs only on feature branches', () => {
    const hook = fs.readFileSync('.husky/pre-commit', 'utf-8');
    expect(hook).toContain('npm run generate-docs-pdf');
    expect(hook).toContain('"$CURRENT_BRANCH" != "develop"');
    expect(hook).toContain('"$CURRENT_BRANCH" != "staging"');
    expect(hook).toContain('"$CURRENT_BRANCH" != "main"');
  });

  test('gitignore blocks PDFs globally but allows doc PDFs', () => {
    const gitignore = fs.readFileSync('.gitignore', 'utf-8');
    expect(gitignore).toContain('*.pdf');
    expect(gitignore).toContain('!README.pdf');
    expect(gitignore).toContain('!docs/*.pdf');
  });

  test('generator has mappings for all doc PDFs', () => {
    const generator = fs.readFileSync('src/services/generate-docs-pdf.ts', 'utf-8');
    for (const doc of ['README', 'RELEASE_NOTES', 'DESIGN', 'SECURITY']) {
      expect(generator).toContain(`${doc}.md`);
      expect(generator).toContain(`${doc}.pdf`);
    }
  });
});
```

## Summary

| Component | What to do |
|-----------|------------|
| `.gitignore` | One `*.pdf` line + `!` exceptions for doc PDFs |
| Pre-commit hook | Generate on feature branches only, `--only` for changed docs |
| Generator script | Map markdown→PDF, support `--only` flag |
| Promotions | No special handling needed — clean merges because one-sided changes |
| Tests | Verify hook guards, gitignore rules, and generator mappings |

## Gotchas Summary

| # | Gotcha | Impact | Fix |
|---|--------|--------|-----|
| 1 | Duplicate `*.pdf` in `.gitignore` | Unpredictable tracking | Single `*.pdf` + exceptions |
| 2 | Missing PDF in hook's `git add` | New docs not staged | Keep hook and generator in sync |
| 3 | Need `git add -f` after gitignore change | PDFs stay untracked | Force-add once, normal after |
| 4 | Hook runs on feature branches, not develop | Confusing at first | Develop gets PDFs via PR merge |
| 5 | PDF generation can fail (no Chrome) | Blocked commits | Non-blocking `if` wrapper |
| 6 | First promotion after setup may conflict | One-time issue | `checkout --theirs` to resolve |
